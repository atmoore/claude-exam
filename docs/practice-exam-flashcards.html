#1Multi-Agent Research SystemError Propagationeasy✗ Missed
The web search subagent times out while researching a complex topic. You need to design how this failure information flows back to the coordinator agent. Which error propagation approach best enables intelligent recovery?
CLICK TO REVEAL ANSWER
#1Multi-Agent Research SystemError Propagation✗ Missed
✓ Correct Answer (A)
Return structured error context to the coordinator including the failure type, the attempted query, any partial results, and potential alternative approaches.
✗ You Chose (D)
Propagate the timeout exception directly to a top-level handler that terminates the entire research workflow.
Why This Is Correct
Structured error context gives the coordinator the information it needs to make intelligent recovery decisions—whether to retry with a modified query, try an alternative approach, or proceed with partial results. Option B's generic status hides valuable context from the coordinator, preventing informed decisions. Option C suppresses the error by marking failure as success, which prevents any recovery and risks incomplete research outputs. Option D terminates the entire workflow unnecessarily when recovery strategies could succeed.

Answer Pattern Cheat Sheet

How to spot the correct answer

The correct answer is almost always the most specific, direct, and actionable option. It names exact mechanisms, flags, file paths, or config options.

Correct patternWrong pattern
"Add the -p flag""Configure non-interactive mode"
"Create rule files in .claude/rules/""Organize your configuration better"
"Check the stop_reason field""Monitor when Claude is done"
"context: fork""Isolate the execution"
"--output-format json and --json-schema""Structure the output"

Wrong answers tend to be:

  • Vague/abstract — "be more specific in prompts", "add better instructions"
  • Overly complex — add a new agent, build a classifier, create infrastructure
  • Workarounds — post-processing, filtering, reducing output (treats symptoms)

More patterns:

  • Built-in over custom — leverage existing API fields (stop_reason, cache_control, system param) over building something new
  • Declarative over imperative — tool_choice: {"type": "auto"} beats "write logic to decide when to call tools"
  • Isolation is the default — subagents get isolated context, not inherited; pass only what's needed
  • Simplest architecture wins — single-agent > multi-agent; hub-and-spoke > mesh
  • Know the anti-patterns — don't parse assistant text for loop control; don't share full history with subagents
  • Caching answers name API features — cache_control breakpoints with ephemeral type, not "reduce token usage"
  • Streaming answers name event types — content_block_delta, message_start, not "listen for updates"
  • Token details matter — input_tokens, cache_creation_input_tokens, cache_read_input_tokens in the usage object
